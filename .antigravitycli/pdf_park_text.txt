프로젝트  소개  
 판매자가  상품과  가격을  초기  설정하고  구매자들이  입찰가를  정하는,  일반적인  경매·중고거래  플랫폼입니다.    본  프로젝트는  채팅과  입찰이  분리되어  발생하는  사용자의  화면  전환  부담을  줄이고자,  채팅방  내부에  경매  탭을  배치했습니다.  이를  통해  사용자는  대화하면서  경매를  참여할  수  있으며,  채팅과  경매  진행  상황을  하나의  화면  흐름에서  관리할  수  있도록  UX 를  설계했습니다.   또한  채팅  내용의  키워드를  분석해  관련  경매·중고거래  상품을  자동으로  추천함으로써,  대화  흐름  안에서  자연스럽게  거래로  이어지도록  설계했습니다.  메시징  구조는  WebSocket·Redis  Pub/Sub  기반으로  구성해  실시간성과  확장성을  함께  확보했습니다.

-  다른  팀원이  먼저  구성한  경매용  Redis  설정에  채팅용  Redis 를  추가하면서  Bean  충돌이  발생했는데,  @Qualifier 로  연결  팩토리를  명시적으로  분리  주입해  해결했습니다.  협업  환경에서는  공통  설정이라도  도메인별로  명확히  구분해두는  것이  유지보수에  유리하다는  것을  체감했습니다.   -  AOP 를  처음  적용하며  '어디까지를  핵심  로직으로  볼  것인가'를  고민했고,  추천  기능을  분리해두니  로직  변경  시  채팅  코드를  건드리지  않아도  된다는  관심사  분리의  장점을  직접  경험하였습니다.   -  채팅방  생성과  방장  멤버  등록이  별개  쿼리로  처리되어  한쪽이  실패하면  방장  없는  방이  생기는  문제가  있었습니다..  @Transactional 로  두  쿼리를  묶어  원자적으로  처리하며,  여러  쿼리가  하나의  작업  단위일  때는  일관성을  코드가  아닌  트랜잭션으로  보장해야  한다는  것을  배웠습니다.   -  채팅  UI  전체를  fetch  API  기반  비동기로  처리하면서  페이지  리로드  없이  동작하는  화면을  만들었습니다.  동기  방식이었다면  메시지  하나,  참여자  갱신  하나마다  페이지가  새로고침되어  채팅  서비스  자체가  성립하기  어려웠것이라고  생각했고,  비동기  처리가  단순한  편의가  아니라  실시간  서비스의  전제  조건이라는  것을  체감했습니다.

성장과정  
 저는  한  번  맡은  일은  책임감을  가지고  끝까지  완성해내는  태도를  중요한  가치관으로  삼고  있습니다.  이러한  가치관이  실질적인  성과로  이어진  경험은  전북지방우정청  인턴  시절에  있었습니다.   당시  매  분기  40개  우체국에서  전달되는  물류  및  수리  현황  데이터를  수작업으로  취합해야  했습니다.  반복  작업이  많아  업무  시간이  오래  걸렸고,  입력  오류가  발생할  가능성도  있었습니다.  저는  단순히  주어진  업무를  처리하는  데  그치지  않고,  이  과정을  자동화해  더  정확하고  효율적인  방식으로  개선해보고자  했습니다.   이를  위해  퇴근  후  VBA  매크로를  독학하며  자동화  로직을  직접  설계했습니다.  우체국마다  데이터  양식이  조금씩  달랐고  예외  상황도  많았기  때문에,  여러  차례  테스트와  디버깅을  반복하며  코드를  수정해  나갔습니다.  단순히  한  번  실행되는  수준이  아니라,  실제  업무에서  안정적으로  사용할  수  있는  형태로  완성하는  것을  목표로  했습니다.   그  결과  40개  우체국의  데이터를  취합하고  검수하는  과정을  자동화할  수  있었고,  업무  속도와  정확도를  높이는  데  기여했습니다.  또한  제가  구축한  자동화  방식은  인턴  종료  후에도  내부  업무에  활용되며  실질적인  개선  효과를  남겼습니다.   이  경험을  통해  저는  문제를  발견했을  때  그대로  받아들이기보다,  더  나은  방법을  고민하고  끝까지  구현해내는  과정에서  실무적  가치를  만들  수  있다는  것을  배웠습니다.  앞으로도  개발자로서  맡은  기능을  단순히  구현하는  데  그치지  않고,  사용자와  팀에  실제로  도움이  되는  방식으로  완성도를  높이는  사람이  되고자  합니다.   
업무  관련  역량

추천 기능을 채팅 전송 로직(ChatService)에 직접 넣을지, 따로 뺄지 고민하다 AOP(@AfterReturning)로 
분리했습니다. 덕분에 추천 로직을 추가·변경할 때 ChatService는 손대지 않고 Aspect만 수정하면 
됐습니다. 서비스에 직접 넣었다면 추천을 고칠 때마다 채팅 핵심 코드를 건드려야 했을 텐데, 
관심사를 분리해 채팅 코드를 단일 책임으로 유지할 수 있었습니다.
캐싱 처리 시 원본 보호 우선
채팅 내역을 Redis에 캐싱하면서, Redis 데이터는 TTL 만료나 메모리 정책으로 언제든 사라질 수 
있다는 점을 전제로 설계했습니다. 그래서 DB를 항상 원본(단일 진실 공급원)으로 두고 Redis는 조회 
속도만 담당하게 했으며, Redis가 비어도 DB에서 다시 읽어 채우도록 만들었습니다. 캐시는 신뢰의 
기준이 아닌 보조 수단이라는 원칙을 코드로 지킨 경험이었습니다.

협업과 Git 형상 관리
Git 형상 관리에 초점을 두고 프로젝트를 진행했습니다. 처음엔 검색 필터(검색어·주제 선택·페이징)를 
한 커밋에 몰아넣었는데, 문제가 생겨 revert하니 정상 동작하던 부분까지 함께 되돌아가 작업을 
처음부터 다시 해야 했습니다. 이 경험을 통해 변경을 작은 단위로 쪼개는 '1기능 = 1커밋' 원칙을 
지키게 됐습니다.
MVC 계층 분리
요청부터 응답까지(Controller->Service->DAO->JSP)흐름을 직접 연결해 보며, 각 계층이 하나의 책임만 
맡도록 나누면 한 곳을 고쳐도 다른 곳에 영향이 가지 않는다는 걸 체감했습니다. 쿼리를 바꾸더라도 화면(JSP)
이나 비즈니스 로직(Service)은 그대로 둘 수 있어, 수정 범위가 좁고 오류가 난 위치 추적도 편해졌습니다.
