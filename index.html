01. 전체 테이블 설계도

초기 요구사항 분석을 바탕으로 설계된 교육센터 운영 시스템의 전체 ERD입니다. 엔티티 간의 복잡한 관계에서 최적의 설계를 찾아내기 위해 노력하였습니다.

02. 협업 및 문서화 도구

프로젝트의 원활한 진행을 위해 Figma를 이용하여 전체 시스템의 흐름과 순서도를 작성했습니다. 또한 Google Sheets를 활용해 팀원 간 업무를 분담하고 진행 상태를 실시간으로 공유했습니다.

03. 테이블 깊이 (Depth) 설계도

테이블 간의 외래키(FK) 참조 관계를 분석하여 데이터 입력 및 삭제 시 의존성 문제가 발생하지 않도록 5단계의 계층 구조(Depth)를 설계했습니다. 이를 통해 순차적으로 더미데이터를 삽입하였습니다.

04. 관계의 재정의

초기 설계에서는 '취업성공수당' 테이블이 '취업상태' 테이블에 종속되어 있었습니다. 하지만 비즈니스 로직상 취업성공수당은 수강생 본인에게 단 한 번만 지급됩니다.

취업상태 컬럼이 변경되는 시점에 수당이 지급되긴 하지만, 일회성 지급이라는 점을 DB단계에서 명확하게 설정할 수 있고 테이블 관계를 변경하더라도 상태컬럼 변경에 따른 지급 로직에는 영향이 없기 때문에 수강생 테이블과 1:1 관계로 재연결하여 DB 구조적 단계에서 일회성 지급 제약을 완벽히 구현했습니다.

05. 정규화 vs 반정규화 고찰

시험 테이블 설계 시, '유형'이라는 상태 코드로 필기/실기를 구분(정규화)하면 NULL을 줄이고 확장에 용이하다는 장점이 있습니다.

하지만 본 시스템의 특성상 새로운 시험 유형이 추가될 가능성이 희박하고, 성적 조회 시 필기와 실기 점수를 동시에 호출하는 경우가 압도적으로 많습니다. 따라서 조회 성능 향상과 쿼리 단순화를 위해 필기/실기 컬럼을 각각 분리하는 반정규화 모델을 최종 채택했습니다.

06. 제약 조건 최적화 (CHECK)

과목별 배점(필기, 실기, 출석)의 합이 100점을 초과하지 않도록 검증하는 로직을 고민했습니다. 초기에는 복잡한 트리거(Trigger)를 사용하여 검증하려 했으나, 트리거 연산이 가져올 오버헤드를 고려했습니다.

결과적으로 테이블 생성 시 CHECK (필기+실기+출석 <= 100) 제약 조건을 명시하는 방식으로 리팩토링하여, DB 성능 최적화와 가독성을 동시에 확보했습니다.

07. PL/SQL: 중도 탈락 처리 로직

출석률 80% 미만인 수강생의 상태를 '중도포기'로 일괄 자동 업데이트하는 프로시저(proc_handle_course_drop)를 구현했습니다.

근태 산정 방식(지각 3회 누적 시 미출석 1회로 환산)을 처리하기 위해 함수(func_calculate_absence)를 만들어 프로시저 내부에서 호출합니다.

08. PL/SQL: 정원 초과 방지 트리거

수강생 등록 시 발생하는 동시성 이슈와 데이터 정합성을 보장하기 위해 정원 확인용 트리거를 설계했습니다.

수강생이 INSERT 되기 직전에 해당 강의실의 정원(Capacity)과 현재 등록 인원을 실시간으로 조회·비교하여, 정원이 초과될 경우 즉시 RAISE_APPLICATION_ERROR를 발생시켜 잘못된 데이터 삽입을 차단합니다.

09. 프로젝트 후기

이번 프로젝트로 SQL, PL/SQL 활용 능력과 ERD 설계 역량을 키울 수 있었습니다. 조인과 서브쿼리에 대한 이해도가 높아졌고, 막연했던 PL/SQL도 직접 구현해보며 자신감을 얻었습니다.

앞으로 배워야 할 방대한 내용에 부담도 느꼈지만, 한 번에 완벽하기보단 꾸준히 성장해야 한다는 다짐을 하게 되었습니다. 의견 차이도 있었지만 서로 소통하고 조율하며 협업의 중요성을 배웠고, 혼자서라면 끝까지 해내지 못할 일도 함께할 팀원들이 있다는 것이 정말 큰 의지가 되었습니다.

01. MVC 아키텍처 및 패키지 구조

JSP Model 2 방식의 MVC 아키텍처를 도입하여 단일 책임 원칙(SRP)을 준수했습니다.

요청 흐름 제어(Controller), 비즈니스 로직(Service), 데이터 접근(DAO) 계층으로 패키지를 명확히 분리함으로써 코드의 결합도를 낮추고 유지보수성을 크게 향상시켰습니다.

02. 스터디 목록: 반응형 그리드와 동적 텍스트 제어

Tailwind CSS 기반의 반응형 카드 UI를 설계하여 다양한 디바이스 환경에 대응했습니다.

특히 사용자가 자유롭게 입력하는 스터디 소개 글이 전체 카드 레이아웃의 높이를 망가뜨리지 않도록, CSS line-clamp 속성을 적용하여 텍스트 길이를 동적으로 제어했습니다. 또한 서버 단의 상태 변수(모집중, 참여중 등)를 JSTL로 평가하여 조건부 뱃지를 렌더링함으로써 클라이언트 스크립트 연산을 최소화했습니다.

03. 스터디 상세: 권한별 분기 및 오작동 방지 UX

서버에서 인가받은 권한 정보(isManager, isMember)를 바탕으로 프론트엔드 노출 버튼(수정, 가입, 탈퇴, 추방 등)을 세밀하게 제어했습니다.

또한 '방장 위임'이나 '멤버 추방' 등 데이터베이스에 치명적인 변경을 가하는 핵심 로직에는 네이티브 confirm()을 활용한 1차 방어선을 구축했습니다. 이를 통해 사용자의 실수로 인한 잘못된 서버 API 호출을 사전에 방지하여 서버 리소스를 보호했습니다.

04. 캘린더 연동: 라이브러리 커스터마이징 및 데이터 바인딩

FullCalendar.js 라이브러리를 연동할 때, 별도의 추가 AJAX 호출 없이 JSTL을 이용해 서버 단의 DTO 리스트를 자바스크립트 이벤트 객체 배열로 즉시 파싱하여 렌더링 속도를 높였습니다.

또한, 기본 캘린더 UI의 막대(Event bar) 패딩이 커서 여러 일정이 겹칠 때 화면이 복잡해지는 문제를 해결하기 위해, CSS 오버라이딩을 통해 이벤트 막대의 높이와 마진을 압축하고 마우스 호버(Hover) 시 시각적 포커싱 효과를 주어 가독성을 크게 개선했습니다.

05. To-Do 리스트: 비동기 통신(AJAX)과 상태 정합성

할 일의 완료 상태를 체크할 때 페이지 전체가 새로고침되어 사용자 경험이 저하되는 것을 막기 위해 Fetch API를 이용한 비동기 통신을 구현했습니다.

브라우저의 즉각적인 체크박스 반응은 허용하되, 실제 완료를 뜻하는 전체 UI(취소선 및 배경색)의 변경은 서버의 DB 업데이트 성공 응답을 받은 직후로 지연시켰습니다. 반대로 서버 통신에 실패할 경우, 자바스크립트로 체크박스 상태를 즉시 원상 복구(Rollback)시켜 클라이언트와 서버 간의 데이터 정합성을 완벽하게 보장했습니다.

// Fetch API 비동기 상태 관리 및 롤백 로직
fetch('/teamtwo/study/todo-status.do', { ... })
.then(res => res.json())
.then(data => {
  if (data.result === 1) {
    // 서버 성공 응답 시 동적 DOM 클래스 추가/제거
    row.classList.toggle('done-row');
  } else {
    // 통신 실패 시 프론트엔드 상태 강제 롤백 방어 로직
    alert('상태 변경에 실패했습니다.');
    this.checked = !this.checked;
  }
});

06. 데이터 무결성 및 관계 모델링

회원과 스터디의 N:N 관계를 해소하기 위해 '스터디 구성원' 매핑 테이블을 설계했습니다.

회원번호 + 스터디번호에 Unique 제약조건을 걸어 중복 가입을 방지하고, 부모 스터디 삭제 시 관련된 할 일 및 구성원 데이터가 고아 데이터로 남지 않도록 ON DELETE CASCADE를 적용했습니다.

07. 논리적 삭제 및 동기화 트리거

서비스 정책 상 회원 탈퇴 시 데이터가 물리적으로 삭제되지 않고 상태(status)만 '탈퇴'로 변경됩니다. 따라서 ON DELETE CASCADE를 활용할 수 없었습니다.

이를 해결하기 위해, 회원의 상태가 변경될 때 트리거를 발동시켜 해당 회원이 가입된 '스터디 구성원' 데이터를 자동으로 DELETE 처리하여 데이터 무결성을 보장하였습니다.

08. 비정규화를 통한 성능 최적화 고찰

가입 시 정원 초과 방지를 위해 조회 및 비교 연산이 포함된 복합 트리거를 설계했습니다. 해당 트리거는 더미데이터 삽입 시 최대정원을 초과하지 않도록 하기위해 설계하였기 때문에, 가입하려는 스터디 리스트들을 전부 수집한 후에 정원초과 여부를 확인하는 복잡한 로직을 갖고 있습니다.

후에 아주 간단하고 강력한 개선방안을 발견하였는데, 스터디 테이블에 '현재 인원수' 컬럼을 추가하는 비정규화를 채택하고 단순 UPDATE 트리거와 CHECK 제약조건으로 로직을 단순화하는 것입니다.

09. 프로젝트 후기

JSP와 Servlet의 동작 원리와 파라미터 전달방법을 숙달할 수 있었습니다. 프로젝트 초기에 명명법을 통일하지 않아 디버깅에 큰 어려움을 겪으며, 사전 설계와 명명법 통일의 중요성을 뼈저리게 느꼈습니다.

어렵게만 느껴졌던 UI/CSS 영역도 직접 다뤄보며 조금 익숙해지게 되었고, Git 브랜치 병렬 작업을 통해 협업 툴을 배울 수 있었습니다. 처음에는 Git 사용법에 익숙하지 않아 큰 어려움을 겪었지만, 익숙해지고 나니 Git없이는 협업이 불가능하다고 느껴질 정도였습니다. 그리고 더미 데이터를 단시간에 생성해야만 했는데, Oracle 고급 문법을 적용해보며 활용할 수 있는 DB문법이 굉장히 많고 깊다는 것을 알게된 프로젝였습니다.

01. 시스템 아키텍처 (System Architecture)

클라이언트의 요청이 WAS(Tomcat)를 거쳐 Spring 컨트롤러로 라우팅되는 전체 흐름도입니다.

일반 HTTP 요청과 WebSocket/STOMP 프로토콜을 분리하여 처리합니다. 특히 고부하 환경에서의 동시성 제어 및 병목 해소를 위해 Redis 인메모리 캐시와 Redisson 분산 락을 설계하였으며, 결제(Toss) 및 이메일(SMTP) 등 외부 API 연동을 포함한 구조로 설계했습니다.

02. 일반 경매 목록: 클라이언트 단 타이머 렌더링

다수의 경매 물품이 동시에 진행되는 목록 화면입니다. 남은 시간을 실시간으로 보여주기 위해 서버에 주기적으로 요청(Polling)을 보내면 엄청난 트래픽 부하가 발생합니다.

이를 해결하기 위해 클라이언트 자바스크립트의 setInterval을 활용하여 브라우저 단에서 독립적으로 남은 시간을 계산하도록 설계했습니다. 마감 1시간 미만 시 텍스트 색상을 붉은색으로 변경하고 깜빡임(animate-pulse) 효과를 주어 사용자의 즉각적인 액션을 유도하는 UI/UX를 구현했습니다.

03. 일반 경매 상세: 프론트엔드 방어 로직 및 부분 렌더링

단일 경매 물품의 상세 정보를 확인하고 입찰을 진행하는 화면입니다. 서버 리소스를 보호하고 사용자 경험을 최적화하기 위해 다음과 같은 클라이언트 단 처리를 구현했습니다.

  • Double Submit 방지: 입찰 버튼 클릭 즉시 disabled = true 처리하여 중복 클릭으로 인한 트래픽 낭비를 차단했습니다.
  • 사전 유효성 검사: 서버 요청 전 브라우저 단에서 금액을 선행 비교하여 무효한 요청을 즉시 걸러냅니다.
  • 웹소켓과 부분 렌더링: 웹소켓(WebSocket)으로는 'auctionSeq'라는 가벼운 데이터만 브로드캐스팅 받아, 실제 갱신할 데이터는 Fetch API를 통해 JSON으로만 받아옵니다. 이후 자바스크립트를 이용해 최고가와 최신 입찰 내역 등 변경이 필요한 DOM 노드만 타겟팅하여 동적으로 렌더링함으로써 사용자 경험을 최적화하고 경매에 참여중인 사용자들이 모두 같은 입찰가를 볼 수 있도록 하였습니다.

04. 실시간 라이브 경매: 동시성 제어 및 캐싱 적용

실시간 스트리밍 방송과 함께 수많은 시청자가 짧은 순간에 동시에 입찰을 시도하는 고부하 환경입니다.

이러한 집중적인 트래픽 상황에서도 동일 가격 중복 입찰을 막기 위해 Redisson 분산 락을 적용하여 데이터 정합성을 철저히 보장했습니다. 또한, 락으로 인한 대기 시간(병목)을 상쇄하고 방송과 입찰 간의 지연을 없애기 위해 Redis 인메모리 캐싱을 도입함으로써 실시간 경매에 걸맞은 고성능 처리를 구현했습니다.

05. 경매마스터 테이블 설계

초기 설계에서는 포인트 락(입찰 시 예치금 락을 걸기위함) 테이블에 경매테이블FK와 라이브경매테이블FK 간에 배타적 참조 제약조건을 두어 무결성을 확보 -> 경매 테이블, 실시간 경매 테이블의 PK역할을 하는 seq가 auction_master의 PK만을 참조하도록 구조를 개선. 이는 아래와 같은 이점을 갖게됨.

  • PK를 상속하는 모양새로써 객체지향적 다형성(Polymorphism) 구현
  • 확장에는 열려있고 변경은 최소화하는 구조로써 OCP원칙 구현
  • 전역 식별성 보장
  • 성능 및 트래픽 최적화 - 마스터 테이블의 PK참조 구조는 단순 조회시에는 이점이 없지만, 경매의 식별자(PK)가 나뉘어 있다면 입찰 기록을 남기거나 Redisson 분산 락을 걸 때 경매 타입에 따라 분기 처리를 하고 락 키를 다르게 관리해야 하는 오버헤드가 발생. 마스터 테이블을 통해 전역 PK를 하나로 통일함으로써, 락 관리 로직을 단순화하고 입찰 트랜잭션 속도 최적화. 동시성 처리 성능을 올리는 방향으로 트레이드오프를 맞추었습니다.

06. 입찰기록 테이블 설계

  • 회원과 경매테이블 간의 다대다 관계를 해소하고 기록을 남기기 위해 입찰기록 테이블 설계
  • @Transactional 사용: @Transactional을 사용하여 두 쿼리를 실행하는 경우 하나의 쿼리만 시행되는 일이 없도록 하여 데이터 정합성을 보장하였습니다.
  • 복합 인덱스 적용: 경매 취소 로직 실행 시 대량의 경매테이블 누적 데이터의 status를 전부 조회하는 경우 발생 가능한 성능 저하를 방지하기 위해 (auction_seq, status) 복합 인덱스를 적용했습니다. 애플리케이션 계층의 쿼리문 수정 없이 오라클 옵티마이저(Optimizer)가 자체적으로 인덱스를 참고하게 됩니다.

07. K6 부하 테스트 1: 동시성 이슈 및 분산 락

K6 부하 테스트(K-1000: 1000명이 0.1초 간격으로 10초간 입찰을 누르는 부하) 중 다중 입찰 시 동일 가격이 중복 입찰되는 이슈를 발견했습니다.

이를 해결하기 위해 Redisson 분산 락을 도입하여 동시 접근 스레드를 제어했습니다. 그 결과, K-5000의 매우 높은 부하 환경에서도 최고가 입찰 로직이 무시되지 않고 지켜지며 데이터 무결성을 확보할 수 있었습니다.

08. K6 부하 테스트 2: 분산 락 병목 및 캐싱 전환

분산 락 도입으로 무결성은 확보했으나, 락 대기 시간(Wait Time) 증가로 인한 전체 응답률 저하 이슈가 발생했습니다.

이를 극복하기 위해 기존 DB를 직접 조회하던 최고가 검증 로직을 Redis 인메모리 캐싱으로 전환했습니다. 락으로 인한 페널티를 인메모리 I/O 속도로 상쇄한 결과, K-1000 부하 기준 응답률이 92.93%에서 98.61%로 상승하였습니다.

09. K6 부하 테스트 3: 최종 성능 개선 지표

Redis 인메모리 캐싱 도입을 통해 매번 DB 디스크를 조회하던 병목 현상을 제거한 최종 결과 지표입니다.

  • 평균 응답 속도: 794.12ms ➔ 2.01ms (약 390배 단축)
  • 초당 처리량(TPS): 1,061건 ➔ 9,593건 (약 9배 증가)
  • 에러율 방어: DB 커넥션 풀 고갈 및 락 대기 타임아웃 감소로 에러율 7.06% ➔ 1.38%로 대폭 하락
  • 응답률 100%을 위해서는 메시지큐의 도입이 필요할 것으로 보입니다. 해당 프로젝트에서는 시간부족으로 도입하지 못하여 아쉬움이 남습니다.

10. 프로젝트 후기

이번 프로젝트는 단순한 구현을 넘어 '시스템 아키텍처 설계'와 '동시성 제어'라는 백엔드의 핵심 과제를 깊이 있게 다룬 경험이었습니다. 특히 분산 락과 인메모리 캐싱을 도입하며 무결성과 응답 속도 사이의 트레이드오프(Trade-off)를 직접 해결하고 성능 단을 대폭 개선했습니다.

그리고 AI 시대의 '개발자의 역할'에 대해 고민해보게 되었습니다. CSS작업과 처음 써보는 Redis 코드 구현에서 Ai가 시간을 많이 단축해준 덕분에 아키텍처 설계와 성능개선에 집중할 수 있었지만, 발생가능한 문제점을 예상하고 변경점에 따른 트레이드 오프(Trade-off)를 결정하고 사람들을 설득하는 것은 결국 개발자의 몫이었습니다. 그렇기에 Ai시대를 살아가는 개발자로써 Ai가 만들어준 의견을 맹목적으로 받아들이는 것이 아니라, 집요하게 의심, 비판하고 학습해야한다는 생각이 들었고, 집요하게 의심하고 질문하는 것은 저의 가장 큰 장점입니다.

"현장의 안전을 책임지던 소방관의 정신으로 시스템의 안정성을 책임지겠습니다."
